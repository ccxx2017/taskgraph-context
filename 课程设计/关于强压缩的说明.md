## 一句话结论

`prompt.ts` 说明：Claude Code 的 Compact 并不是“随便让模型做个摘要”，而是一个**强约束、强结构化、明确禁止工具调用**的摘要流程，核心目标是：

1. **强制纯文本输出**
2. **先写 `<analysis>` 草稿，再写 `<summary>` 正式摘要**
3. **摘要必须覆盖 9 个固定栏目**
4. **支持全量 compact 与 partial compact 两套语义**
5. **最终会把 `<analysis>` 删除，只保留格式化后的 summary**
6. **compact 后注入后续会话的不是原始 XML，而是一个包装后的 continuation message**

## 对“第三幕：Compact 系统深潜”里相关表述的源码级修正

下面我只说和 `prompt.ts` 直接相关、最值得修正或加强的点。

### 1. 之前说“包含9个部分的结构化摘要”——这个判断是对的，而且现在有源码实锤

从 `BASE_COMPACT_PROMPT` 可直接确认，完整 compact summary 要求包含这 **9 个 section**：

1. Primary Request and Intent
2. Key Technical Concepts
3. Files and Code Sections
4. Errors and fixes
5. Problem Solving
6. All user messages
7. Pending Tasks
8. Current Work
9. Optional Next Step

这不是“报告作者归纳出的九部分”，而是 **prompt 模板硬编码要求**。

---

### 2. “禁止工具调用”不是普通提醒，而是非常激进的首部约束

这点比一般理解更强。

源码里有一个专门的：

```ts
const NO_TOOLS_PREAMBLE = `CRITICAL: Respond with TEXT ONLY. Do NOT call any tools.
...
- Tool calls will be REJECTED and will waste your only turn — you will fail the task.
- Your entire response must be plain text: an <analysis> block followed by a <summary> block.
`
```

而且末尾还有一层：

```ts
const NO_TOOLS_TRAILER =
  '\n\nREMINDER: Do NOT call any tools...'
```

也就是说它不是只在结尾提醒一次，而是：

- **开头强警告**
- **正文模板要求**
- **结尾再提醒**

这是一个典型的 **prompt sandwich** 结构，目的是降低模型误触发工具调用的概率。

你前面教案里说了 `NO_TOOLS_PREAMBLE`，这个方向没错；但如果要更精准，可以强调：

> 这不是一般性的“请不要调用工具”，而是为了应对某些模型在单轮 compact 中误调用工具、导致整轮摘要失败而设计的**强抑制提示**。

---

### 3. `<analysis>` 的作用比“分析块”更明确：它是 drafting scratchpad

源码注释写得很明确：

```ts
// The <analysis> block is a drafting scratchpad that
// formatCompactSummary() strips before the summary reaches context.
```

以及：

```ts
// Strip analysis section — it's a drafting scratchpad that improves summary
// quality but has no informational value once the summary is written.
```

所以更准确的表达不是只是“剥离分析块，保留摘要内容”，而是：

- `<analysis>` 是**为了提升摘要质量**的中间草稿区
- 它在进入后续上下文前会被删除
- 真正进入持续会话上下文的是 **格式化后的 summary**

这点很重要，因为它体现了 Claude Code 在 compact 上的一个设计思想：

> **允许模型在压缩时“多想一点”，但不把这些思考成本永久带入后续上下文。**

---

### 4. `formatCompactSummary()` 不只是“提取 `<summary>`”，还会把它改写成普通文本头

源码实际做了三步：

#### 第一步：删除 `<analysis>...</analysis>`

```ts
formattedSummary = formattedSummary.replace(
  /<analysis>[\s\S]*?<\/analysis>/,
  '',
)
```

#### 第二步：提取 `<summary>...</summary>` 内容，并替换成：

```ts
Summary:\n${content.trim()}
```

也就是说最终不是保留 XML 标签，而是转换成普通可读文本。

#### 第三步：压缩多余空行

```ts
formattedSummary = formattedSummary.replace(/\n\n+/g, '\n\n')
```

所以如果你要修正文案，建议把“剥离分析块，保留摘要内容”升级成：

> `formatCompactSummary()` 会删除 `<analysis>` 草稿区，提取 `<summary>` 内容，并将其规范化为以 `Summary:` 开头的纯文本摘要。

这个说法更贴近源码。

---

### 5. 你前面的教案里说“摘要作为用户消息注入，而不是修改 system prompt”，这点从本文件看是成立的

这里关键函数是：

```ts
export function getCompactUserSummaryMessage(...)
```

函数名本身已经说明它生成的是 **user summary message**。

而且它包装成这样的文本：

```ts
This session is being continued from a previous conversation that ran out of context. The summary below covers the earlier portion of the conversation.
```

然后拼上格式化后的 summary。

这直接支撑了前面教案中的判断：

- compact 后摘要不是写进 system prompt
- 而是作为一条继续会话用的用户态上下文消息注入

这是个很重要的架构点，你前面的解释是对的。

---

### 6. `getCompactUserSummaryMessage()` 比教案里讲得更丰富：它还有三种续接控制

这个是前面教案没展开、但源码里很值得补充的点。

这个函数有几个控制参数：

#### `transcriptPath?: string`

如果传入，就会追加一句：

> 如果需要 compaction 前的具体细节，可以去读完整 transcript

这说明 compact 后系统并不假装“摘要等于全部信息”，而是保留了一个**回查原始转录**的出口。

#### `recentMessagesPreserved?: boolean`

如果为 true，会追加：

```txt
Recent messages are preserved verbatim.
```

这说明 compact 后可能存在一种模式：

- 旧消息被摘要
- 最近消息原样保留

这和你前面报告/教案里提到的“保留最近消息组，摘要较早消息组”是互相吻合的。

#### `suppressFollowUpQuestions?: boolean`

如果为 true，会给模型一段非常强的 continuation instruction：

- 不要承认 summary
- 不要 recap
- 不要说 “I’ll continue”
- 像中断从未发生一样直接继续最后任务

这说明 compact 后续接不仅是“塞一个摘要”，而且包含 **会话连续性伪装/恢复策略**。

这个点很适合写进课程，因为它非常体现 agent engineering 的味道。

---

### 7. Partial compact 不是一个模板，而是三种语义中的两种“局部模式”

源码里至少有三套 prompt：

- `BASE_COMPACT_PROMPT`
- `PARTIAL_COMPACT_PROMPT`
- `PARTIAL_COMPACT_UP_TO_PROMPT`

其中 `getPartialCompactPrompt(customInstructions, direction)` 里：

- `direction === 'from'` → `PARTIAL_COMPACT_PROMPT`
- `direction === 'up_to'` → `PARTIAL_COMPACT_UP_TO_PROMPT`

这意味着你前面如果只是泛泛说“partial compact”还不够精确。更准确地说：

### Partial compact 有两种语义

#### 1) `from` 模式
总结“保留上下文之后的 recent messages only”。

也就是：
- 更早的 retained context 还在
- 你只需要总结后面新增的 recent portion

#### 2) `up_to` 模式
总结“当前看到的这段完整历史”，并且这个总结将被放在后续 session 的开头，后面还会接上更新的消息。

源码注释写得很清楚：

```ts
// 'up_to': model sees only the summarized prefix (cache hit). Summary will
// precede kept recent messages, hence "Context for Continuing Work" section.
```

这说明 `up_to` 模式明显是服务于一种：

- 前缀摘要
- 最近消息保留
- 摘要前置到继续会话开头

的上下文拼装策略。

这部分如果你后面要继续完善“第三幕”或“完整生命周期”，非常值得加进去。

### 可直接确认的结论

- Compact prompt 采用 **analysis + summary 双层输出结构**

  - 指向 transcript
  - 标记 recent messages 已保留

## 这个文件最体现工程味道的 4 个点

我认为最值得在课程里强调的是这 4 个：

1. **强 no-tools 双保险**  
   开头 preamble + 结尾 trailer，防止单轮 compact 被工具调用浪费掉。

2. **允许分析，但不保留分析**  
   用 `<analysis>` 提升摘要质量，再在进入上下文前删除。

3. **摘要不是终点，而是 continuation payload**  
   `getCompactUserSummaryMessage()` 说明 compact 是为“继续工作”服务，不只是为了省 token。

4. **partial compact 是有方向语义的**  
   不是简单“局部摘要”，而是服务于不同上下文拼接策略。
