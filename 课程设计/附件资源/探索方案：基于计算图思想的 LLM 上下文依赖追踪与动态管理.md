# 探索方案：基于计算图思想的 LLM 上下文依赖追踪与动态管理

## 1. 灵感来源与背景 (Inspiration & Background)

### 1.1 核心灵感：Micrograd 与 `Value` 类
本方案的直接灵感来源于 Andrej Karpathy 的 **Micrograd** 项目。在构建微分引擎时，Karpathy 并没有仅仅将数字视为标量，而是通过 `Value` 类赋予了数字“记忆”：
- **`self.data`**: 存储数值。
- **`self._prev`**: 显式记录前驱节点（谁创造了它）。
- **`self._op`**: 记录操作类型（如何创造的）。
- **`trace()` 函数**: 能够从输出端逆向遍历整个计算图。

这种设计使得**自动微分**成为可能：系统不仅知道结果，还清楚结果的“血缘关系”，从而能精确计算梯度。

### 1.2 问题映射：从“数值微分”到“逻辑推理”
当前的大语言模型（LLM）在推理阶段面临类似但不同的挑战：
- **现状**：LLM 将上下文视为线性的 Token 流或压缩的 KV Cache。模型依靠隐式的注意力机制（Attention）关联信息，缺乏显式的逻辑依赖记录。
- **痛点**：
    1. **黑盒性**：无法解释结论具体依赖哪段历史。
    2. **上下文浪费**：滑动窗口策略会无差别丢弃旧信息，可能误删关键前提。
    3. **更新困难**：修改一处事实往往需要重算整个上下文，缺乏局部性。

### 1.3 核心假设
如果我们借鉴 Micrograd 的思想，将 LLM 的推理过程建模为**动态依赖图（Dynamic Dependency Graph）**：
- 让每个推理步骤（或语义块）成为一个“节点”。
- 显式记录该步骤依赖的前文片段（`_prev`）。
- 记录推导逻辑（`_op`，如“总结”、“反驳”、“引用”）。

那么，我们有望实现：
- **可解释的推理链**：像 `draw_dot` 一样可视化 AI 的思考路径。
- **智能上下文剪枝**：基于图遍历识别并移除“孤立节点”（无关历史），保留关键依赖。
- **增量更新**：仅重算受影响的子图，提升长文本处理效率。

## 2. 探索目标 (Objectives)

1.  **验证可行性**：证明让 LLM 显式输出依赖关系是可行的，且不会过度增加延迟。
2.  **构建原型工具**：开发一个能够解析 LLM 输出、构建依赖图并进行可视化的 Python 工具（类似 `draw_dot` 的文本/逻辑版）。
3.  **评估优化效果**：对比“基于图的上下文管理”与传统“滑动窗口”在长文本问答中的准确率与 Token 消耗。
4.  **教学转化**：将成果转化为编程教学案例，帮助学生理解“数据结构如何决定算法能力”。

## 3. 技术路线与实施步骤 (Roadmap)

### 第一阶段：概念验证 (PoC) - “手动思维链图谱”
*   **周期**：2 周
*   **任务**：
    1.  **Prompt 工程设计**：设计 Prompt，要求 LLM 在回答复杂问题时，以结构化格式（JSON）输出每一步推理及其依赖源 ID。
        *   *示例输出*：`{"step_id": 3, "content": "因此...", "depends_on": [1, "input_segment_A"]}`
    2.  **数据类构建**：仿照 `Value` 类，编写 `ReasoningNode` 类，包含 `content`, `prev_nodes`, `operation_type`。
    3.  **图构建器**：编写解析器，将 LLM 的 JSON 输出转换为内存中的图结构。
    4.  **可视化验证**：利用 `graphviz` 渲染推理图，人工检查依赖关系是否合理。
*   **成功标准**：能够生成清晰的推理流程图，且依赖逻辑符合人类直觉。

### 第二阶段：工具化与自动化 - “AI 推理调试器”
*   **周期**：3-4 周
*   **任务**：
    1.  **流式解析**：适配 LLM 的流式输出（Streaming），实时构建图节点。
    2.  **自动依赖提取**：尝试不依赖硬编码 Prompt，而是通过分析 Attention 权重或向量相似度，自动推测 `_prev` 关系（作为辅助验证）。
    3.  **交互界面**：开发一个简单的 Web 界面（Streamlit/Gradio），左侧显示对话，右侧实时渲染动态生成的思维图谱。
    4.  **错误注入测试**：人为修改输入中的某个事实，观察图谱中哪些节点被标记为“受影响”。
*   **成功标准**：形成一个可用的演示 Demo，能直观展示 AI 的“思考过程”。

### 第三阶段：优化策略验证 - “基于图的上下文 GC”
*   **周期**：4-6 周
*   **任务**：
    1.  **模拟长对话**：构建一个超过模型 Context Window 限制的超长对话场景。
    2.  **实现剪枝算法**：
        *   定义“活跃节点”：当前问题直接或间接依赖的节点。
        *   定义“垃圾节点”：未被任何活跃节点引用的历史节点。
        *   执行剪枝：移除垃圾节点，仅保留活跃节点及其必要摘要。
    3.  **对比实验**：
        *   组 A：传统滑动窗口（FIFO）。
        *   组 B：基于图的动态剪枝。
        *   指标：回答准确率、Token 节省率、推理延迟。
*   **成功标准**：在保持准确率不显著下降的前提下，证明组 B 能更有效地利用有限的 Context Window。

## 5. 风险评估与应对 (Risk Management)

| 风险点 | 描述 | 应对策略 |
| :--- | :--- | :--- |
| **模型配合度低** | LLM 可能无法稳定输出正确的依赖关系 JSON。 | 1. 使用 Few-Shot Prompting 提供高质量示例。<br>2. 引入“校验步骤”，让另一个小模型专门检查依赖关系的合理性。 |
| **性能开销大** | 维护图结构和解析 JSON 可能增加推理延迟。 | 1. 仅在关键推理步骤（CoT）建图，普通闲聊跳过。<br>2. 以“语义块”而非“Token”为节点粒度，降低图规模。 |
| **依赖提取不准** | 自动推测的依赖关系可能存在噪声。 | 将此作为“辅助建议”而非绝对真理，允许人工修正，重点在于验证“显式图”这一架构的价值。 |
| **理论偏差** | 逻辑依赖不完全等同于数学微分，直接套用可能导致误解。 | 在文档和教学中明确区分：这是**“因果依赖图”**而非**“微分计算图”**，避免概念混淆。 |
