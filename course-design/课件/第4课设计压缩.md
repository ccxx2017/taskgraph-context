
# [COMPACT]
## 修正后的整体课程设计压缩（基于后续定位修正）

### 课程总定位
**《从 Claude Code 到 Logic Graph：构建依赖感知的上下文管理器》**  
不再定位为单纯解读源码或还原 AI 思考，而是：**从 Claude Code 暴露的 Agent 工程线索出发，学习上下文管理、压缩、功能演化，并原创实验系统 Logic Graph——任务态上下文管理管线**。

**核心主线修正**：Claude Code 展示工业级应对上下文爆炸的手段；Logic Graph 不是通用压缩器，而是**任务态依赖索引与上下文结构化机制**。最终形成管线：  
`对话 → Graph Patch → TaskGraph → Context Pack → 主LLM`（以及 Memory 沉淀、多 Agent 协作、证据追踪、调试报告）。

---

### 课程结构（10讲，每讲定位已按后续修正）

#### 第0讲：导论——Claude Code 事件真正值得学的是什么
- 44个 feature flags 是理解系统演化的入口，上下文管理是核心战场。  
- 引出课程双线：Claude Code 的工程方法 + Logic Graph 的实验性回应。  
- 核心结论：Claude Code 暴露的是 Agent 工程方法论，而非单个功能。

#### 第1讲：门控体系——复杂 Agent 如何管理功能演化
- 六层门控（编译期、环境变量、远程实验、配置文件、API 能力、Kill Switch）。  
- 以 COORDINATOR_MODE 等为例追踪真实门控链路。  
- 迁移设计：为 Logic Graph 设计五层门控（建图、调试、GC、可视化、实验特性）。  
- 结论：门控是管理演化复杂性的架构能力。

#### 第2讲：上下文工程——Claude Code 如何在有限窗口中持续工作
- 上下文预算模型（固定开销 + 可变开销 + 净空间）。  
- 五层防御体系：源头控制 → 折叠压缩(Context Collapse) → 微压缩(MicroCompact) → 全量压缩(Auto Compact) → 应急压缩(Reactive Compact)。  
- 摘要强结构化（9个固定栏目）+ `<analysis>` 草稿区 + 禁止工具调用双保险。  
- 对 Logic Graph 启示：梯度响应、摘要作为消息、断路器、按语义单元分组；图结构可实现比无差别摘要更智能的压缩。

#### 第3讲：从 Micrograd 到 Logic Graph——上下文不是文本，而是依赖图
- Micrograd 的 `Value` 类（`_prev`, `_op`）启发依赖追踪。  
- Logic Graph 定位修正：**不是还原模型内部思维，而是记录任务中外部可验证依赖**。  
- 节点类型：UserGoal, Constraint, Fact, ToolResult, Decision, Summary, FinalAnswer。  
- 核心价值：回答“哪些旧信息不能丢”“前提变化后哪些结论失效”“工具错误后哪些答案受影响”。

#### 第4讲：让任务变成图——构建多轮对话的增量 Logic Graph（写入侧）
- **本讲定位**：用户对话 → Graph Patch → TaskGraph，回答“任务图怎么长出来”。  
- 任务级图 vs 轮次级图：一个任务一张图，每轮只做增量 Patch（new/updated/superseded）。  
- 启动时机：手动（如 `/task start`），后续可半自动/全自动。  
- 典型操作：通过 Graph Extractor Prompt 让 LLM（旁路小模型）输出结构化 Patch，Python 代码合并成 TaskGraph。  
- 最小可运行代码：Node, Edge, TaskGraph, GraphUpdater, Deduplicator, to_mermaid。  
- 核心原则：不删除被推翻节点，用 `superseded` 留下演化轨迹。

#### 第5讲：从 TaskGraph 到 Context Pack——把任务图喂回主 LLM（读取侧）
- **本讲定位修正**：不是压缩，而是 **TaskGraph → Context Pack → 主LLM**，回答“任务图怎么喂回去”。  
- Context Pack 是结构化任务上下文（current_focus, goals, active_constraints, current_decisions, relevant_facts, open_tasks, superseded_history），不是图 dump。  
- 从当前焦点（如最新 active OpenTask）向上游追踪依赖边（`depends_on`/`implements`/`serves`/`refines`），同时收集被 `invalidates` 边标记的旧方案。  
- 最小代码实现 `build_context_pack` + `context_pack_to_prompt`，注入主 LLM。  
- 核心价值：让主 LLM 明确知道当前有效约束、已推翻方案及原因，避免重蹈覆辙。

#### 第6讲：任务记忆沉淀——从 TaskGraph 到长期 Memory Store
- **本讲定位修正**：不是“压缩升级版”，而是跨任务、跨会话的**长期任务记忆沉淀与召回**。  
- 流程：任务阶段结束 → Memory Extractor 抽取长期记忆 → Memory Store；新任务时召回 → 生成 MemoryReferenceNode 进入新 TaskGraph → Context Pack。  
- 沉淀类型：GoalMemory, ConstraintMemory, PreferenceMemory, DecisionMemory, OpenTaskMemory, ErrorMemory, ArtifactMemory（可选）。  
- 不沉淀：临时推理、大段工具输出、已推翻方案、闲聊。  
- 生命周期：create, update, merge, invalidate, archive, delete。失效标记而非删除，留痕供后续参考。  
- 召回后通过 `memories_to_seed_nodes` 接入 TaskGraph，不走直接塞 prompt。

#### 第7讲：多 Agent 协作——TaskGraph 的边界与验证（重写版）
- **本讲定位**：解决多 Agent 时 TaskGraph 的一致性、边界性和可验证性。  
- 核心架构：Coordinator 基于 TaskGraph 为每个 Worker 生成 **Worker Context Pack**（子任务上下文）；Worker 只返回 **Result + Graph Patch**（不直接改图）；Coordinator 合并 Patch、处理冲突；Verifier 独立验证关键 Claim + Evidence，返回 Verification Patch。  
- 三种包：Worker Context Pack, Worker Result Package, Verification Pack。  
- Demo 延续 Minecraft：拆分 3 个 Worker → 生成 Context Pack → 返回 Patch → 合并时发现客户端命令与无 OP 权限冲突 → Verifier 验证 RCON → 更新 TaskGraph → 重新生成主 LLM Context Pack。  
- 核心原则：TaskGraph 单写入，Worker 多读取；广播式上下文爆炸被消除。

#### 第8讲：工具与文件系统——让工具结果成为可追踪的证据节点
- **本讲定位**（中等调整）：工具调用、文件产物、测试结果作为可追踪、可压缩、可失效的上下文证据。  
- 四类核心节点：ToolCallNode, ToolResultNode, FileArtifactNode（含版本哈希）, TestResultNode。  
- 大结果写盘 + 预览引用（preview + full_ref），连接第5讲压缩（可丢 preview，不可丢被依赖的 full_ref）。  
- 失效传播简化版：当工具结果错误或文件变化，标记 `tainted` 向下游传播。  
- Demo：根据 config.yaml 写代码 → 测试 → 修改 config.yaml → 下游节点变 tainted/suspect/invalid。  
- 核心结论：信息有来源，依赖有方向，文件有版本，结果能回查。

#### 第9讲：可观测性与调试——把 TaskGraph 管线变成可检查的 Debug Report
- **本讲定位重写**：不是“AI Reasoning Debugger”，而是 **TaskGraph 管线的工程调试器**。  
- 核心问题：系统输出错了，是 Extractor 抽错、Graph 合并错、Context Pack 漏、Memory 召回错，还是证据失效？  
- 六类可观测对象：Patch, TaskGraph Snapshot, Context Pack, Memory Recall Report, Evidence Index, Event Timeline。每类必须包含 `reason` 字段。  
- **本讲最重要部分**：Context Pack Trace——为 Pack 中每个条目增加 `included_reason` / `excluded_reason`，解释为何入选/排除。  
- 产出静态 **Task Context Debug Report**（含 conversation.md, patches.json, task_graph.json, context_pack.md, memory_recall.json, evidence_index.json, timeline.json, graph.mmd）。  
- 实时 UI 作为选做扩展。

#### 第10讲：总结与路线图——从课程原型到真实项目
- **本讲定位重写**：闭环前9讲形成完整工程管线；明确 Logic Graph“是什么、不是什么”；给出后续路线。  
- **明确定位**：Logic Graph 是**任务态上下文管理机制**，不是模型内部推理还原，不是通用聊天压缩器，不是 Compact 替代品，不是万能长期记忆，不是可上线的工业系统。  
- **最终 Demo（Minecraft 全流程）**：  
  用户提出任务 → Extractor 输出 Patch（目标/偏好/事实）→ 补充约束（一个键、无 OP 权限）→ Patch 新增 Constraint，标记旧客户端方案 superseded，新增服务端 RCON 决策 → Context Builder 生成 Context Pack（目标、有效约束、当前方案、已推翻方案）→ Memory 沉淀 → 新会话召回记忆 → 工具证据入图（读取配置、测试 RCON）→ Debug Report 解释为何建议 RCON 而非客户端命令。  
- **后续路线图**：  
  - 工程：封装 Python 包（extractor / graph / context_builder / memory / coordinator / evidence / debug_report / examples）。  
  - 产品：做成“Task Context Debugger for AI Agents”，卖点为主 LLM 可见上下文、漏掉约束、旧方案被推翻、工具证据支撑、多 Agent 合并。  
  - 研究：抽取准确率、Context Pack 对长任务完成率提升、Memory 过期冲突、多 Agent Patch 一致性。  
  - 教学：可打包成 AI Agent 工程课 / 上下文管理专题课。  
- **最终收束**：课程真正主线不是 Claude Code 的某个 flag，而是**复杂 Agent 系统如何管理任务状态**。Claude Code 展示了工业手段，Logic Graph 尝试回答：若显式记录目标、约束、决策、证据和被推翻方案，Agent 能否更稳定地继续工作。

### 标题：第4讲教案：Logic Graph第一阶段PoC——让对话变成图

**本讲定位**：从概念进入可运行原型的第一讲。将普通对话自动/半自动生成结构化元数据，再转成Logic Graph节点和边，形成最小闭环。

**核心目标**（学生理解三点）：
1. 不要让模型输出完整思维链，而是提取目标、约束、事实、工具结果、决策和待办。
2. Logic Graph第一阶段只记录“任务元数据”（目标、约束、事实、工具结果、决策、修改文件、未完成任务）。
3. 先做能跑的PoC：对话 → 结构化JSON → 节点和边 → Mermaid图可视化。

**标题建议**：第4讲：让对话变成图——构建Logic Graph的第一个可运行原型；副标题：从自然语言任务到结构化上下文元数据。

**课时**：90分钟（可压缩为60分钟）。

### 教学结构

**0. 开场（5分钟）**：Logic Graph是Agent工作过程的依赖索引，第一步是把普通对话里的关键上下文抽出来。板书：普通对话 → 结构化元数据 → 节点 → 边 → 图。

**1. 定义元数据Schema（15分钟）**：
- 最小schema：
```json
{
  "user_goals": [],
  "constraints": [],
  "facts": [],
  "tool_results_used": [],
  "decisions": [],
  "files_changed": [],
  "open_tasks": []
}
```
- 字段含义：用户目标、约束、事实、所用工具结果、决策、修改文件、未完成任务。
- 强调：不使用“Reasoning Step”，只使用Goal/Constraint/Fact/Decision/ToolResult/FileArtifact/OpenTask。

**2. LLM抽取提示词（15分钟）**：
- 让LLM做结构化提取，不输出推理过程。提供标准Prompt（角色限定为元数据抽取器，严格输出JSON，不输出Markdown，按规则填充各字段）。
- 价值：限制输出范围 → 减少幻觉 → 可解析 → 方便生成图。

**3. 课堂示例（15分钟）**：展示一段对话（AI课程设计需求），期望输出示例JSON（包含目标、约束、事实、决策、文件改动、待办等）。

**4. 最小可运行代码（25分钟）**：单文件`logic_graph_poc.py`。
- 定义Node(id, type, content)、Edge(source, target, relation)、LogicGraph类（add_node, add_edge, to_mermaid）。
- `build_graph_from_metadata(metadata)`：创建根节点Task，依次添加user_goals→UserGoal节点并连边，constraints→Constraint，facts→Fact，tool_results_used→ToolResult，decisions→Decision（并为每个Decision添加从目标/约束/事实/工具结果来的影响边），files_changed→FileArtifact（并从Decision指向它），open_tasks→OpenTask（并从Decision指向它）。
- 主程序：使用示例metadata调用，输出Mermaid代码，复制到Mermaid Live查看。

**5. 课堂练习（15分钟）**：提供Minecraft AI Agent对话（中文指令、孩子能用、无OP权限、建议RCON）。要求学生：①提取JSON元数据；②填入代码；③运行生成Mermaid图。给出期望JSON结构。

### 老师反复强调的关键点

- 第一版图不需要完美，目的是证明“对话可结构化、结构化可成图、图可辅助压缩”。

### 标题：第4讲的多轮任务图修正方案

用户指出：第4讲教案只处理一轮对话的依赖图，但实际任务需要多轮对话，一轮对话无法反映全貌。用户质疑：何时开启依赖图？每轮都建图会导致大量重复；启动是手动还是自动？这些问题如何解决？

GPT-5.5 确认质疑正确，提出核心修正：不应每轮创建新图，而是一个任务创建一张 **Task-Level Logic Graph**，每轮对话只生成 **Graph Patch**（增量更新），通过 Upsert 避免重复（节点存在则更新，不存在则新增，旧节点被推翻则标记 superseded）。

**开启时机**：建议三种方式——手动启动（用户明确指令，适合教学）、半自动启动（系统检测复杂目标/文件/代码/连续推进等条件后提示用户确认，更适合真实产品）、全自动启动（高级阶段，需要任务识别与相似度判断）。

**用户操作**：用户只需控制“开始任务”“切换任务”“结束任务”，中间每轮系统自动做增量更新。

**每轮输出**：不输出完整图，只输出 Graph Patch（包含 task_id、new_nodes、updated_nodes、new_edges、superseded_nodes）。

**第4讲修改方案**：分两阶段——阶段一用一轮对话生成初始图作为教学演示（但必须声明这不代表真实系统）；阶段二立即升级为“已有 Task Graph + 新一轮对话 → 抽取 Patch → 合并”。最终架构包含 TaskRouter、GraphPatchExtractor、NodeDeduplicator、GraphUpdater、GraphStore。

**标题修改建议**：原“让对话变成图”改为“让任务变成图：构建多轮对话的增量 Logic Graph”。

**核心原则**：一个任务 = 一张持续更新的图；每轮对话 = 对图的一次增量补丁。

### 标题：第4讲教案修订版——让任务变成图：构建多轮对话的增量 Logic Graph

**本讲定位**：一个任务 = 一张持续更新的图；每轮对话 = 一次 Graph Patch。目标不是造完美系统，而是让学生跑通最小多轮依赖图 PoC。

**核心目标**：学生能回答：何时开始建图（识别到任务才建）？每轮做什么（只生成增量补丁，不重建图）？重复节点怎么办（Upsert + superseded 标记）？

**课时**：90分钟。

**教学结构**：开场回顾与问题抛出（5分钟）→ 任务级图 vs 轮次级图（10分钟）→ 何时启动依赖图（10分钟）→ Graph Patch 数据结构（10分钟）→ 典型操作示例：三轮任务演化（20分钟）→ 最小可运行代码（25分钟）→ 课堂练习（10分钟）。

---

### 0. 开场回顾（5分钟）
指出每轮建新图的问题：一轮没有全貌、大量重复、启动时机不明、图管理混乱。结论：一个任务一张图，每轮一次补丁，补丁只记录变化。

---

### 1. 任务级图 vs 轮次级图（10分钟）
错误模型：Turn1→Graph1, Turn2→Graph2（大量重复）。正确模型：TaskGraph 持续存在，每轮产生 Patch（new_nodes / updated_nodes / superseded_nodes）合并进图。价值不在建图，而在演化。

---

### 2. 何时启动依赖图（10分钟）
- **模式 A 手动启动（教学用）**：用户指令 `/task start 设计AI课程`，系统创建 TaskGraph。
- **模式 B 半自动启动（推荐产品）**：系统检测到复杂目标/文件/连续推进等信号时询问用户。
- **模式 C 全自动启动（高级）**：自动判断短问答不建图、持续任务自动建图等。本讲只实现模式 A。

---

### 3. Graph Patch 数据结构（10分钟）
- **节点结构**：`node_id, type, content, created_at_turn, status`（active/superseded/done）。
- **Patch 结构**：`task_id, turn_id, new_nodes[], updated_nodes[], superseded_nodes[], new_edges[]`。
- **节点类型**：UserGoal, Constraint, Fact, ToolResult, Decision, FileArtifact, OpenTask（七种）。

---

### 4. 典型操作示例：三轮任务演化（20分钟）⭐
**场景**：构建 Minecraft AI Agent，让孩子用中文控制游戏。

**第1轮**：用户输入目标（中文控制 Minecraft）。系统创建 TaskGraph，Patch1 新增 UserGoal×2、Fact×1，边 `refines`。图状态：Task + UserGoal(n_0001, n_0002) + Fact(n_0003)。

**第2轮**：用户要求写命令映射器、客户端斜杠命令执行。系统挂接现有任务，Patch2 新增 Decision×2，边 `serves` 和 `implements`。去重器避免重复创建已有 UserGoal。

### 标题：Logic Graph的运行时架构：旁路监听与LLM抽取

**用户问题**：是否有另外的程序监听用户与主LLM的对话，并根据规则调用另一个大模型来生成Logic Graph？

**核心结论**：理解基本正确，可拆分为三种架构：

1. **旁路监听+独立LLM抽取**（最常见、最解耦）：主对话LLM正常回答用户；旁路Graph Extractor程序监听对话，调用另一个（通常更便宜的）小模型抽取JSON格式的Graph Patch，再由GraphUpdater合并进TaskGraph。优点：主对话不受影响，抽取失败不干扰用户体验；缺点：多一次LLM调用，成本约为主对话30%-60%。这是第4讲教案隐含的架构。

2. **主LLM自己顺手输出Patch**：通过function calling/structured output，让主模型在回答用户的同时额外输出结构化JSON。优点：省一次LLM调用，成本低；缺点：主模型分心可能影响回答质量，对模型结构化输出能力要求较高。

3. **混合架构**（真实产品最常见）：轻量Patch由主LLM顺手输出，复杂Patch由旁路LLM批量抽取，推翻/冲突检测等用规则程序处理，不走LLM。原则：能用规则不用LLM，必须语义理解才用LLM。

**直接回答三个隐含问题**：
- 是否有另一个程序监听对话？是，叫Graph Extractor/Orchestrator。
- 它是按规则还是调用LLM？两者都有：结构化合并（GraphUpdater、Deduplicator）靠规则，语义抽取靠LLM。
- 抽取一定要用“另一个”LLM吗？不一定：可用独立小模型（省钱、解耦），也可让主模型顺手输出（省一次调用）。

**第4讲代码对应位置**：教案代码只实现了规则部分（GraphUpdater + Deduplicator），未实现LLM抽取。完整系统应增加`GraphPatchExtractor`模块，其`llm_client`可为另一个便宜模型、主模型再调一次或本地小模型。

**一句话总结**：旁边确实有个“监工程序”在监听对话，它用规则+另一个（通常更便宜的）LLM将每轮对话变成Graph Patch，再合并进任务图；主对话LLM只负责回答用户，图的事它可以完全不知道。

### 标题：Logic Graph 偏移原因分析：Extractor 输入契约不完整

用户发现执行两轮后生成的 TaskGraph 存在偏移（如重复开工单节点、风险节点孤立等），询问是否为方案设计问题。

**核心结论**：`logic_graph_task.py` 合并代码基本没问题，偏移主要源于第4课方案设计中 **Extractor 输入契约不完整**，导致 LLM 生成 patch 时发生语义偏移。

**五个具体原因**：

1. **Extractor 缺少当前完整 TaskGraph 作为输入**  
   每轮仅给了用户原话，未提供已有图状态和当前 turn_counter。LLM 不知道哪些节点已存在、应 superseded 或已 active，导致重复创建具体化节点（如两个开工单任务）。

2. **抽取范围不明确**（抽取用户原话还是助手回答）  
   图中出现了许多源自助手规划建议的内容（如时间线、阶段计划、成功标准），若将其当作已确定 Decision，会导致图偏离用户真实任务。建议课堂 PoC 严格限定：**只抽取用户原话**，除非用户明确确认。

3. **节点类型不足**  
   限定七种类型（UserGoal, Constraint, Fact, ToolResult, Decision, FileArtifact, OpenTask），但任务中出现了风险、成功标准、时间线、验收标准等，LLM 只能硬塞入 Fact，表达力不够。建议增加 `Risk`、`SuccessCriteria`、`Timeline` 类型。

4. **status 状态不一致**  
   课件写 `active/superseded/done`，Prompt 写 `active/superseded/resolved`，代码用 `resolved`。应统一为 `active/superseded/resolved`，不用 `done`。

5. **代码只做精确去重，不做语义重复判定**  
   `type + content` 完全一致才去重，不同字符串的同一任务不会自动标记 superseded 或补边。语义上的具体化、替代、补边必须由 LLM 在 patch 中明确输出（如 `superseded_nodes` 和 `refines` 边）。

**对第4课的核心修正建议**：  
从第2轮开始，Extractor 的输入必须包含：
- 当前完整 TaskGraph（不是上一轮 patch）
- 当前 turn_counter
- 本轮用户原话
- 输出要求：只生成 `turn_id = turn_counter + 1` 的 patch

即：每轮必须将最新的 `graph_state.json` 作为上下文提供给 LLM，否则无法生成正确的增量 patch。

**最终判断**：不是合并代码的问题，而是 Prompt/流程设计缺少“当前完整 TaskGraph 作为输入”的强约束。建议优先修改课件，而非合并代码。
### 标题：偏移原因修正与第4课Prompt补充规则

用户针对GPT-5.5之前的五点原因诊断提出反驳：1) Graph抽取内容应为完整一轮对话（用户+助手），而非仅用户输入；2) 节点类型应保持固定数量，不能为每次新词增加类型，否则失去泛化能力；3) 用户已手动将当前TaskGraph粘贴给Extractor，并非缺失；4) status不一致易修改；5) 语义重复问题后续处理。GPT-5.5接受前三点，修正诊断：偏移的根本原因不是“只抽用户输入”“必须增加节点类型”或“缺少当前TaskGraph”，而是**第4课的Extractor Prompt缺少三类关键规则**：

1. **完整一轮对话抽取规则**：Logic Graph应同时抽取用户输入和助手输出，否则无法记录任务实际推进（方案、决策、待办等）。Prompt需明确“抽取完整一轮对话，但只抽取与持续任务相关的目标、约束、事实、决策、产物、待办和工具结果”。

2. **固定节点类型的语义映射规则**：不随意扩张类型（如Risk、SuccessCriteria等），而是用固定类型通过content和relation表达语义。例如：风险/限制/不可违反条件 → Constraint；已知背景/观察结果 → Fact；已确认方案/阶段安排 → Decision；待办 → OpenTask；文件产物 → FileArtifact。保持泛化能力。

3. **基于当前TaskGraph的旧节点对齐规则**：Prompt必须强制优先执行：①检查已有active节点 → ②优先使用`updated_nodes`更新 → 其次`superseded_nodes`标记废弃 → 再次`new_edges`连接 → 最后才`new_nodes`。且除根UserGoal外，每个新增节点必须至少有一条边，避免孤立节点。

具体案例：当用户从“开工单TKT-2026-XXX”细化到“开工单TKT-2026-002”时，不应创建新节点，而应更新原节点内容或将其标记superseded并建立refines边。风险节点若影响执行边界归为Constraint，若仅为观察描述归为Fact。

**最终结论**：Logic Graph总体方案和合并代码没有问题，偏移来自第4课Extractor Prompt的规则不完整。补全上述三类规则后，偏移将显著减少。
### 标题：节点生命周期判定规则：从枚举到泛化

用户指出：GPT-5.5 此前给出的“典型情况”（如 TKT-2026-XXX 变为 TKT-2026-002）属于具体枚举，无法泛化到其他场景。

GPT-5.5 接受批评，认为应补充的不是“典型情况列表”，而是**抽象的节点生命周期判定规则**。

**核心规则**：Extractor 在新增节点前，必须将本轮每个候选信息与当前 TaskGraph 中所有 active 节点对齐，从四种动作中选择一种：

1. **SAME_ENTITY_MORE_DETAIL（同一对象补充细化）** → 使用 `updated_nodes` 更新原节点，不新建。
2. **REPLACES_OLD_ENTITY（替代旧对象）** → 旧节点放入 `superseded_nodes`，新建节点，用 `invalidates` 或 `refines` 边连接。
3. **DEPENDS_OR_SUPPORTS_EXISTING（依赖/支撑已有对象）** → 新建节点，必须用 `new_edges` 连接到已有节点。
4. **NEW_ENTITY（真正新对象）** → 仅在无法归入前三类时新建节点，除根 UserGoal 外必须至少有一条边。

**禁止项**：
- 因措辞变化而创建重复节点。
- 同一任务/决策/约束同时存在多个 active 版本。
- 创建没有边的孤立节点（根 UserGoal 除外）。

**结论**：规则不应枚举具体场景（如工单号、阶段计划等），而应基于抽象关系判断，才能具备泛化能力。第4课 Prompt 应升级为上述生命周期判定规则。
